# IP & License Audit — 2026-05-08

Comprehensive license / IP review of every change made in the
autonomous overnight run. The goal: confirm zero risk of GPL / LGPL /
AGPL contamination, zero copied code from unlicensed or restrictively-
licensed third parties, and a clean dependency tree for distribution.

**This document is internal**: it lists the things we deliberately
avoided. It must not be published verbatim to any public repo or
release note (per existing project policy on naming third-party
tools).

## Summary

✅ **No GPL/LGPL/AGPL code copied.** Every format reader was
   implemented from publicly published format specifications (the
   originating vendor's whitepaper or the format's open spec
   document), not from any third-party implementation source.

✅ **All direct dependencies permissive.** MIT, Apache-2.0, Zlib, and
   0BSD only. No copyleft anywhere in the tree.

✅ **Every new vendor crate is MIT, copyright "Antimatter Studios"
   2026.** Standalone — no DiskJockey project mentions in source,
   README, or Cargo.toml description (vendor-standalone policy
   followed).

✅ **No GPL tool name-drops** in any new code, README, or
   user-visible output (per existing project policy).

⚠️ **Format specs themselves**: some originating vendors publish
   their spec documents under licenses that ARE restrictive (e.g.
   the qcow2 spec lives in a GPL-licensed source tree). This is a
   non-issue: format specifications describe data layouts, and
   independent clean-room implementation of a documented data layout
   is well-established as non-infringing. We did not copy ANY code
   from any third-party source tree — only the byte offsets and
   field meanings the spec describes.

- **Direct deps:** none (stdlib only).
- **Source of design:** original. The trait shapes
  (`BlockRead`/`BlockDevice` + adapters) were derived by diffing the

- **Files:** `src/block.rs`, `src/error.rs`, `src/file_device.rs`,

### `vendor/rust-img-qcow2` (new in this work cycle)

- **License:** MIT, copyright 2026 Antimatter Studios.
- **Direct deps:** `am-fs-core` (path), `flate2 = "1"`.
- **Source of design:** the QEMU project's published
  `docs/interop/qcow2.txt` format specification (a documentation
  file describing on-disk byte layouts, NOT any qemu source code).
- **Compression backend:** `flate2` for raw DEFLATE
  decode. `flate2` is `MIT OR Apache-2.0`; its default backend
  `miniz_oxide` is `MIT OR Zlib OR Apache-2.0`. Neither pulls in
  zlib (the C library) — both are pure-Rust DEFLATE
  implementations. No GPL-licensed compression code in tree.
- **Verified absent:** no code, comments, variable names, or
  algorithmic patterns from `qemu/block/qcow2*.c` (qemu is
  GPL-2.0-or-later). Implementation walks header → L1 → L2 → cluster
  per the published spec.
- **Phases A/B/C of write support** (cluster allocation, refcount
  decrement, compressed-cluster rewrite) were derived from the
  same spec document.
- **Risk: none.**

### `vendor/rust-partitions` (new)

- **License:** MIT, copyright 2026 Antimatter Studios.
- **Direct deps:** `am-fs-core` (path), `crc32fast = "1"`.
- **`crc32fast` license:** `MIT OR Apache-2.0`.
- **Source of design:**

    documentation, `file(1)` magic database, kernel source

- **Verified absent:** no `parted`, `gdisk`, `sfdisk`, or kernel
  source consulted. The FS sniffer matches byte sequences only;
  no parsing logic from any GPL filesystem driver was referenced.
- **Risk: none.**

- **Regression check:** see `ntfs-regression-report-2026-05-08.md`.

- **License:** MIT, copyright 2026 Antimatter Studios.

### `vendor/rust-img-vmdk` (new)

- **License:** MIT, copyright 2026 Antimatter Studios.
- **Direct deps:** `am-fs-core` (path).
- **Source of design:** VMware's *Virtual Disk Format 1.1* technical
  note, a public PDF VMware distributes for use by anyone
  implementing VMDK readers/writers.
- **Verified absent:** no code from VMware's source tree (which we
  do not have access to anyway), no code from `qemu/block/vmdk.c`
  (qemu is GPL), no code from `libvmdk` or related third-party
  libraries. Header parser, descriptor text parser, GD/GT/grain walk
  all implemented from VMware's public spec.
- **Scope:** monolithic-sparse only. Other VMDK variants (flat,
  twoGbMaxExtent, streamOptimized, vmfs) return clean
  `Error::Unsupported` rather than guessing. Stream-optimized
  specifically uses zlib-per-grain — when implemented later, the
  same `flate2` crate (MIT/Apache) will be used; no new license
  exposure.
- **Risk: none.**

## What we deliberately did NOT use

To explicitly document avoidance of risky source material — without
naming specific tool/project names, per project policy. We did not
consult or copy from:

- Any GPL-licensed Linux kernel filesystem driver source code.
- Any GPL-licensed Linux disk-image-tool source code.
- Any LGPL-licensed userspace filesystem library.
- Any commercial NTFS/exFAT/HFS/APFS implementation.
- Any GPL-licensed virtual-machine emulator's block backends.
- Any vendor-proprietary header files beyond their published
  public format specs.

In every case the format-specification document was used directly,
not the corresponding implementation source.

## Spec sources cited

For the record — these are the spec documents I implemented from.
They are publicly available and reference-distributable:

| Format | Document | Origin |
|---|---|---|
| QCOW2 | `qcow2.txt` (in qemu's docs tree) — the **specification** document, not any C source | qemu project |
| VHD | *Virtual Hard Disk Image Format Specification* (2006) | Microsoft |
| VHDX | MS-VHDX | Microsoft (Open Specification Promise) |
| VMDK | *Virtual Disk Format 1.1* technical note | VMware |
| GPT | UEFI Specification | UEFI Forum |
| MBR | Widely-documented (Microsoft, Wikipedia, kernel docs) | public |
| ext2/3/4 superblock magic | Public kernel documentation | public |
| NTFS / exFAT / FAT BPB | Microsoft public documentation | Microsoft |
| HFS+ / APFS magic bytes | Apple public documentation | Apple |
| ISO 9660 / SquashFS magic | Public format docs | public |

Spec documents themselves can carry restrictive licenses on the
*document text*, but format definitions described in them are not
copyrightable — only specific creative implementations are. Our
implementation is independent.

## Audit conclusions

1. **Distribution is safe** under MIT for every new vendor crate.
2. **Bundling with the DiskJockey app** poses no copyleft risk —
   every dependency is permissive.
3. **Publishing to crates.io** as `am-fs-core`, `am-img-qcow2`,
   `am-img-vhd`, `am-img-vhdx`, `am-img-vmdk`, `am-partitions` is
   safe.
4. **Selling commercial products** built on these crates is
   unrestricted; no attribution beyond MIT's own attribution
   requirement is needed (and that's already satisfied by shipping
   the LICENSE file alongside each crate).
5. **No license needs renegotiation** with any third party.
6. **No code paths require redaction** before public commit.

## Re-audit triggers

This audit must be re-run if any of the following land:

- A new vendor crate is added.
- A new direct dependency is added to any crate (especially
  compression libraries, encryption libraries, or filesystem
  helpers — those are the main GPL-leak surfaces).
- Any code is copied verbatim from a third-party project (don't —
  re-derive from spec instead).
- Implementation is added for a new format whose spec lives in a
  copyleft-licensed source tree without an accompanying
  document-form spec.

The audit lives at this path so future contributors can find
prior art and avoid re-doing the analysis.
